# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Chromium extension (Manifest V3) that enhances SAS Studio 3.8 (a legacy Dojo 1.x web app) by monkey-patching in the page's MAIN world. Two main features: toggling SAS Studio's built-in editor for a modern Ace editor at runtime, and various independent UX fixes (formerly a Tampermonkey userscript, now fully absorbed into the extension — nothing to install separately anymore). Everything works by reverse-engineered runtime patching — no build step or bundler for the extension's own code (`./build_lib.sh` only generates the gitignored `lib/`), and `test/smoke.js` is the test suite.

A live SAS Studio instance for testing runs at `http://192.168.1.72/SASStudio/38/`.

## Components

Everything below lives in one extension, all patching the same app. The repo root is the extension root ("Load unpacked" points at it): `manifest.json` sits at the root, all source files (`sw.js`, `relay.js`, `editor-swap.js`, `ss-fixes.js`, `tools-meta.js`, `defaults.js`, `ace-patches.js`, `popup.html`/`popup.js`, `options.html`/`options.js`, and the custom `ace/` files) live under `src/`, and `assets/` (icons) + `lib/` (gitignored, generated by `./build_lib.sh` — vendored Ace/ace-linters byte-identical to the npm tarballs plus the built SAS LSP bundle; see `src/ace-patches.js` for why nothing under it is ever hand-edited) stay at the root. Consequences of that split: manifest entries and `chrome.scripting` `files:` lists are extension-root-relative (`src/...`), while `options.html`/`options.js` reference the Ace lib page-relatively as `../lib/...`. `./package.sh` zips exactly `manifest.json src assets lib` into `dist/` (gitignored), running `./build_lib.sh` first if `lib/` is incomplete and leaving out the dev-only rest (docs, `test/`, `SASStudio-3.82/`).

- `sw.js` — service worker, three jobs (no `chrome.commands` handler anymore — the editor toggle / native-mouse toggle / browse / command-palette are all in-page `ss-fixes.js` actions with rebindable hotkeys, e.g. editor toggle defaults to `Ctrl+.`, plus the popup's buttons; the only browser command is `_execute_action`, which Chrome itself handles to open the popup, unbound by default). In-page editor toggles post `{ __ssextBadge }` → `relay.js` → a `chrome.runtime.onMessage` listener here that sets the per-tab `ON`/`OFF` toolbar badge (MAIN-world code can't call `chrome.action`; the popup sets its own badge directly):
  1. ss-fixes injection: on every `tabs.onUpdated` for a `/SASStudio/` URL, injects `tools-meta.js` + `ss-fixes.js` into the MAIN world and calls `window.__ssf.init(settings)` with the persisted patch/hotkey settings from `chrome.storage.local`. The same handler pre-injects `editor-swap.js` (+`ace-patches.js`) and seeds `window.__ssExt.libPath`/`userSnippets`/`aceConfig` (the latter via `getAceConfig()`, storage merged over `DEFAULT_ACE_CONFIG`) so the global command-palette / browse hotkeys and any editor created on that page load already have what they need.
  2. Live snippet apply: on `chrome.storage.local`'s `snippets` key changing, pushes the new text into every open SASStudio tab via `window.__ssExt.applySnippets`.
  3. Live ace-config apply: on `chrome.storage.local`'s `aceConfig` key changing (written either by options.html or, via `relay.js`, by the in-page settings panel), merges it over `DEFAULT_ACE_CONFIG` and pushes it into every open SASStudio tab via `window.__ssExt.applyAceConfig`.
- `relay.js` — ISOLATED-world content script (`*://*/SASStudio/*`, `manifest.json`'s `content_scripts`), the only bridge from MAIN-world code to `chrome.storage`: listens for `window.postMessage({ __ssextAceConfig: config })` (source-checked) and writes it to `chrome.storage.local.aceConfig`. Nothing else.
- `editor-swap.js` — defines `AceEditorAdapter` (full reimplementation of the `SAS.Editor` API on top of Ace) and the `window.__ssExt` singleton (`active`, `activate`, `deactivate`, `toggle`, `loadNewAce`, `browse`, `commandPalette`, `applySnippets`, `applyAceConfig`, `libPath`, `aceConfig`). `commandPalette(libPath)` (libPath optional once `ssExt.libPath` is already known — see the global hotkey path below) serializes through the same `_pending` chain as `toggle`/`browse`: it determines the currently-focused Ace editor *before* opening anything (`focusedAceEditor()` — checks `_textViewers` first, then `appDMS.tabs.getAllTabObjects()` code tabs, via each candidate's `.isFocused()`), then `loadNewAce()`s and calls `openCommandPalette(focusedEditor)` with that editor (or `null` if nothing was focused). The adapter's own in-editor `openCommandPalette` command (Alt-Shift-P) calls the same function with `this.aceEditor`. `openCommandPalette` is built on the stock, vendored `ace/ext/prompt` module (`lib/ace/src-noconflict/ext-prompt.js`) rather than a custom prompt module: it builds an entries array — always `SS-Ext: <label>` entries (one per `SSF_TOOLS` action — which now includes `toggleEditor` and `toggleNativeMouse`, so no special-casing) and, only when a focused editor was passed, that editor's own commands (walking `editor.keyBinding.$handlers`, deduped by name, mirroring `prompt.commands`'s `getEditorCommandsByName`, excluding `["insertstring","inserttext","setIndentation","paste"]`) — plus a side-table of runner functions keyed by each entry's `command` field (needed because `prompt.commands`'s `getCompletions` JSON-clones the entries on every filter, which would strip functions), then calls `ace.require("ace/ext/prompt").prompt(focusedEditor || null, "", { name: "commands", selection: [0, Number.MAX_VALUE], onAccept, getPrefix, getCompletions })`, copying `getPrefix`/`getCompletions` filtering from that same preset. A global in-page hotkey (default Alt+Shift+P, rebindable like any other `ss-fixes.js` action — see the `commandPalette` entry in `SSF_TOOLS`) opens the palette even with Ace not activated and nothing focused: `sw.js`'s `tabs.onUpdated` handler pre-injects `editor-swap.js` and seeds `ssExt.libPath`/`ssExt.userSnippets` on every SASStudio page load, so the no-arg `window.__ssExt.commandPalette()` call has what it needs to load the Ace lib on demand. The popup's "Command palette…" button is an alternative entry point, passing an explicit `libPath` (there is no chrome command for it anymore — the only browser command is `_execute_action`, which opens the popup). On first injection it installs one-time dispatcher patches: `SAS.Editor` becomes a function returning either an `AceEditorAdapter` or the original editor depending on `__ssExt.active`, and `DMSEditor.prototype.createCodeEditor` is wrapped the same way (original saved on `__ssExt`, guarded by `_aceReplacementPatched`). `loadNewAce()` loads the newer Ace library from `lib/ace/src-noconflict/` once (its CSS stays attached permanently — SAS Studio renders its own editor DOM via EditorView.js and never references any `.ace_*` class, so the vendored ace's own styles are inert either way). `activate()`/`deactivate()` swap only the `window.ace` global between the app's original bundled Ace and the newer one — that global still matters because runtime `ace.require(...)` calls (Mode.js/SyntaxColorerAdapter.js) resolve against it for tokenization — then walk every open tab converting its editor in place and re-binding `textChanged`/`selectionChanged`/`caretMoved`. Undo history does not survive a toggle in either direction. `browse(kind, libPath, snippetsText)` (used by `window.__ssExt.browse` for browse_ss prompts) only calls `loadNewAce()`, not `activate()` — it works without activating the editor replacement. `applySnippets(text)` registers the user's custom Ace snippets (`ace/snippets` snippetManager, additive over the built-in set) — no-op until the new Ace lib is loaded. `appDMS.createFileView` is also wrapped once (`_createFileViewPatched`): while active, the "View file as text" viewer gets a read-only `AceEditorAdapter` overlay (registry in `__ssExt._textViewers`) laid over the real `SimpleTextarea`, which is kept alive in the widget tree (just visually hidden) so AppDMS's positional load/refresh code (`AppDMS.js:3931`/`3985`, which indexes `pane.getChildren()[1].getChildren()[0]` rather than going through `tabHolder.simpleTextArea`) keeps working untouched; the widget's `.set("value", ...)` is patched to also mirror writes into Ace. The overlay is always editable; a real edit marks the tab dirty (`*name` in the tab title, matching `DMSEditor.applyChangedIndicationToTab`) and enables saving. Save is wired three ways — a toolbar `Save` button (TXT/LOG only), `Ctrl/Cmd+S`, and vim `:w`/`:wq`/`:x` — all POSTing the edited text to the workspace endpoint (mirrors `DMSEditor.saveFile`'s plain-file path); vim `:q`/`:wq`/`:x` also close the tab. The vim ex-commands are registered once on the shared `ace/keyboard/vim` module and resolve the acting `cm.ace` back to either a text-viewer entry or a code-editor tab, so they work in both. `deactivate()` disposes the adapter, clears the dirty marker, unhides the textarea, and restores its original `.set`; viewers opened while inactive are unaffected. (Note: `loadNewAce()` overrides the module URL for `ace/keyboard/vim`/`emacs`/`sublime`/`vscode` to `keybinding-<name>.js` — this build otherwise resolves them to a 404, which had silently disabled non-Ace keyboard handlers entirely.) Persistent editor config: `ssExt.aceConfig` (`{ darkTheme, lightTheme, options, vimrc }`, seeded by `sw.js` and refreshed by `applyAceConfig(config)`) backs `getAceConfig()`, which the adapter constructor reads for its initial theme/`ace.edit()` options, and which `applyConfig(cfg)` (an adapter method) re-applies live (theme via the existing matchMedia machinery — `this._darkTheme`/`_lightTheme` are read live by the matchMedia handler, not captured at construction — plus `aceEditor.setOptions(cfg.options)`). `applyAceConfig(config)` stores it on `ssExt.aceConfig`, calls `applyConfig` on every open adapter (code tabs + `_textViewers`), and (if `config.vimrc` changed since the last apply) reapplies vimrc — see below. `sw.js` calls it on `aceConfig` storage changes. There is no custom preferences panel: Ace's own stock settings menu (`showSettingsMenu`, default `Ctrl-,`/`Cmd-,` command, also surfaced generically in the command palette's per-editor commands list whenever one is focused) is THE settings UI. Loading `ext-settings_menu.js` (in `loadNewAce()`, replacing what used to be a separate `ext-options.js` load — that file bundles its own copy of `OptionPanel`, so eagerly loading it makes the stock command's lazy `config.loadModule("ace/ext/settings_menu", ...)` a no-op, not a second HTTP load) is enough for the menu to work at all; `installSettingsMenuPersistence()` (called once per `loadNewAce()`, guarded by `ssExt._settingsMenuPatched`) additionally patches `ace.require("ace/ext/options").OptionPanel.prototype.setOption` so panel changes stick: after the original runs (which already fires the panel's own `"setOption"` signal), if `option.path` isn't `"theme"`/`"mode"` it updates `ssExt.aceConfig.options[option.path]`, live-applies via `applyAceConfig`, and `window.postMessage({ __ssextAceConfig: cfg }, "*")`s it for `relay.js` to persist (theme is options-page-only, see below; "mode" isn't part of the persisted config). `options.js` installs the identical prototype hook (guarded by its own flag) for the snippet editor's settings menu, writing straight to `chrome.storage.local.aceConfig` instead of postMessage-ing.

SAS language server (LSP): `ensureLsp()` lazily starts one shared ace-linters `LanguageProvider` for the whole page, memoized on `ssExt._lspStarting` (a failure sets `ssExt._lspFailed` so it's never retried until reload). It bails to `null` if `getAceConfig().lsp === false` (the options page's "SAS language server" checkbox, `aceConfig.lsp`, default on); otherwise it derives the extension root from `ssExt.libPath` (same suffix-strip pattern as `srcAcePath`), `fetch(..., { method: "HEAD" })`-probes `lib/sas-lsp/sas-server.js` (console.warn + `null` if the gitignored bundle hasn't been built by `./build_lib.sh`), loads `lib/ace-linters/language-client.js`/`ace-language-client.js` as classic scripts (temporarily deleting the global `window.define` around the load — Dojo's own AMD loader satisfies the UMD wrapper's `typeof define === "function" && define.amd` check and would register the module into Dojo's registry instead of setting `window.LanguageClient`/`window.AceLanguageClient`), then constructs the worker as `new Worker(URL.createObjectURL(new Blob(["importScripts(...)"])))` pointed at the server bundle (blob+`importScripts`, not a fetched string, to avoid holding the ~22 MB bundle in memory as JS source) and calls `window.AceLanguageClient.for(serverData, { functionality: { completion: { overwriteCompleters: false }, semanticTokens: true } })`. Two load-bearing workarounds ported from `SASStudio-3.82/sas-lsp-demo`: (1) ace-linters 2.2.0's `filterByFeature` checks `capabilities.hoverProvider == true`, but the SAS server advertises the object form, so hover silently never fires — a `Object.defineProperty(worker, "onmessage", ...)` shim (an `addEventListener` wrapper sees the message too late) coerces object-form `hoverProvider`/`documentHighlightProvider` to `true` in the initialize response and sets `ssExt._lspReady = true`; (2) with `overwriteCompleters: false` ace's own completers stay registered alongside the LSP one, and the popup sorts by score, so each default completer's `getCompletions` is wrapped to subtract `1e6` from every result's score so LSP entries sort first. `AceEditorAdapter`'s constructor calls `_maybeRegisterLsp()`, which gates on `_lspEligible()` — mode must be `ace/mode/sas` AND (`aceConfig.lspMaxLines <= 0` or the session's current line count is within it, default 500, configurable in the options page next to the "SAS language server" checkbox) — before calling `ensureLsp()` (fire-and-forget `.then`, guarded by `_lspRegistering` so a second call while the promise is already in flight is a no-op); eligibility is re-checked right before `provider.registerEditor` inside the `.then`, in case a big `setText()` landed while it awaited. It registers the created Ace editor on the resolved provider (also guarded by a `this._disposed` flag set in `dispose()`, in case the adapter was torn down before the promise settled), and schedules the demo's 2-second post-register semantic-token kick (`languageProvider.$getSessionLanguageProvider(session).getSemanticTokens()`, try/catch — the initial request races the server's `didOpen`). The code editor is constructed with empty content and its real text arrives later via `setText()` (text viewers/toggle conversion pass content at construction instead) — `setText()` re-runs the eligibility check after setting the value: if now registered-but-over-the-limit it unregisters (`ssExt._lspProvider.unregisterEditor(this.aceEditor, true)`, clears `_lspRegistered`, logs one `[SS Ext]` line); if not registered and now eligible it calls `_maybeRegisterLsp()` again. Typing growth past the limit after that isn't monitored (a `// ponytail:` comment on the spot notes it — a reload or another `setText` picks it up). `dispose()` calls `ssExt._lspProvider.unregisterEditor(this.aceEditor, true)` (closes the document server-side) before `aceEditor.destroy()`. The worker/provider persist for the whole page lifetime once started — deactivating/reactivating the Ace toggle just re-registers editors against the same one, nothing tears it down.

Vimrc: `aceConfig.vimrc` is a small subset of vim config lines (`map`/`nmap`/`imap`/`vmap`, `noremap` variants, `unmap` variants — one mapping per line, `"` comments, unsupported lines just `console.warn` and get skipped) applied to the shared `ace/keyboard/vim` module's `Vim.map`/`Vim.noremap`/`Vim.unmap`. `editor-swap.js`'s `applyVimrcLine(Vim, line)` is the parser (duplicated, not shared, in `options.js` as the same function — MAIN-world and options-page code can't share a file); `applyVimrcConfig(text)` (editor-swap.js) and `installVimExCommands()`'s vim-module-loaded callback both drive it, tracking `ssExt._vimrcApplied` (a counter, for test visibility) and `ssExt._vimrcLastText` (to skip a no-op reapply in `applyAceConfig`). Options.js applies the same parser to the snippet editor's vim handler on init and on `aceConfig` storage changes, with no such tracking (nothing tests it there). Removing a line from the vimrc doesn't undo that mapping until the page reloads — `Vim.unmap` isn't run automatically for a diff, only for whatever the current text still asks to unmap.
- `ss-fixes.js` — various independent SAS Studio UX fixes/features (tab management, tree navigation, keyboard shortcuts, clipboard, context menus), split into `ACTIONS` (one-shot commands, e.g. `reloadCurrentFile`) and `PATCHES` (passive monkey-patches applied once at init, e.g. `keepAlive`). Exposes `window.__ssf = { init(settings), run(name) }`; `init()` waits for `.dijitTreeNode`, applies enabled patches, runs any action `setup()`s, and binds hotkeys; `run(name)` invokes a single action on demand (used by the command palette's `SS-Ext:` entries). The dominant pattern for patches is wrap-and-delegate: save the original method (e.g. `tabs.closeTab`), replace it with a wrapper that calls through. `minimizeBusyDialog` wraps `appDMS.dialogs.postBusyDialog` (the modal run-progress dialog `DMSEditor.submitHandler` posts) and, when the created dialog is the foreground-run dialog (a cancel callback was passed — only the submit paths do), auto-minimizes it on the spot (no button, no click; the options-page checkbox for this patch is the only config — unchecked leaves SAS Studio's dialog untouched; non-run busy dialogs like "Reading file" keep stock modal behavior). Minimizing calls `dijit.Dialog._DialogLevelManager.hide(dialog)` to release the shared modal underlay early (safe — `Dialog.prototype.destroy()` calls the same function again when the run ends, and its out-of-order-removal branch no-ops for an already-popped stack entry), overrides the instance's `_position` to a no-op, and pins `domNode` (220px wide) into the bottom-right corner via computed `top`/`left` — not `right`/`bottom` anchors, which would make dijit's title-bar drag stretch the box instead of moving it (the minimized dialog stays draggable). The single-run guard is two-layered, installed at auto-minimize time, i.e. run start (a code tab, and so a `DMSEditor` class to grab off `tabs.getAllTabObjects()`, is guaranteed to exist then): (1) a prototype wrap of `DMSEditor.prototype.submitHandler`/`interactiveSubmitHandler` short-circuits with a yellow ss-ext busy notice (`#ssf-busy-notice`, a top-left fixed in-page element — SAS Studio's own toaster truncates longer messages, and `sendClientNoteMessage` only reaches the hidden message log) whenever `appDMS.dialogs.busyDialog` is already non-null — but pre-existing tabs' Run button/F3 were wired with `dojo.hitch(this, this.submitHandler)` at tab construction (DMSEditor.js:4234/5017), capturing the ORIGINAL function, so the wrap only covers tabs created after install (plus the interactive path, which the original resolves dynamically through the prototype at DMSEditor.js:6499); so (2) auto-minimize also walks every open tab and `set("disabled", true)`s each not-already-disabled `submitButton` — the original `submitHandler` bails on `this.submitButton.get("disabled")` (DMSEditor.js:6502, SAS Studio's own F3 guard), so this blocks both the click and F3 in pre-existing tabs with no re-wiring. Background submits are deliberately left alone (button untouched, handler unwrapped): `backgroundSubmitButton` is hitched to `submitBackgroundHandler` (DMSEditor.js:5026/6353), an independent method posting to its own `/submissions` endpoint in a separate SAS session, and SAS Studio supports several at once — only the foreground run is single-run. Only the buttons recorded as disabled-by-us are re-enabled at run end, via run-once instance wraps of the dialog's `hide` and `destroy` (success tears down via `hideBusyDialog()`→destroy, the error path via `submitDialog.hide()`); buttons the app disabled itself (e.g. read-only tabs) are never touched. The patch also wraps `dojo.xhr` (all `dojo.xhrGet`/`xhrPost` calls delegate to it at call time): a request whose URL contains `/workspace/` fired while `dialogs.busyDialog` exists gets the busy notice (throttled 3s, non-warn styling) saying it's queued until the run finishes; the notice is sticky (click-to-dismiss) and cleared automatically when the busy dialog closes (its `hide`/`destroy` are wrapped in the `postBusyDialog` wrapper, minimized or not) — it describes a state that's true exactly as long as the run lasts, so no timeout. Run-in-progress detection is `runActive()` — the run's own dialog (marked in the `postBusyDialog` wrapper when a cancel callback is passed, which only the submit paths do) checked for `.open && !._destroyed` — NOT `dialogs.busyDialog != null`: every run-end path tears down via `submitDialog.hide()`, never `hideBusyDialog()`, so `dialogs.busyDialog` stays a stale non-null reference after every completed run. `appDMS.handleWebOneEvent` is also wrapped: while `runActive()`, action `"FileOpenWithTextViewer"` (always) and `"FileOpen"` of a TXT/LOG/LST file (the text-view branch's exact type set; `getFileType` honors the `openWithType`/`fileType` overrides) are refused with a warn notice. The block must sit at that entry point — the open chain fires SYNCHRONOUS xhrs against the busy session (`getModifiedTime`, AppDMS.js:3424, `sync: true`), freezing the entire JS thread until run end (a notice written just before the sync call doesn't even paint), and the text branch (AppDMS.js:3921-3936) additionally posts an UNCANCELABLE "Reading file" modal whose posting destroys the minimized run dialog. All as-text entry points call `appDMS.handleWebOneEvent` dynamically at click time (tree context menu DMSProjects.js:1746, tree double-click DMSProjects.js:3191, file shortcuts DMSFiles.js:667, browse_ss `openItemInSs`), so the instance wrap is never bypassed by construction-time hitches; `appDMS.perspectiveFileOpen` carries the same TXT/LOG/LST check as a backstop for chains that skip `handleWebOneEvent`. SAS-type files take the editor branch and stay allowed (empty tab fills at run end) — those requests hit the same single-threaded workspace session that's executing the program, so the server answers them only at run end, which with the dialog minimized would otherwise look like a hang (e.g. a newly opened file's tab staying empty).
- `tools-meta.js` — shared `SSF_TOOLS` array (`{name, kind, label, title, hotkey}`), one entry per `ss-fixes.js` action/patch. Plain script (no modules) loaded both as an extension-page `<script>` (popup/options) and injected into the page's MAIN world alongside `ss-fixes.js`.
- `defaults.js` — shared `DEFAULT_SAS_SNIPPETS` (the custom snippet text, additive over ace's own built-in SAS snippets) and `DEFAULT_ACE_CONFIG` (`{ darkTheme, lightTheme, options: {...}, vimrc: "", lsp: true, lspMaxLines: 500 }` — the persisted-config default; storage wins per-key, shallow merge of the top level and of `options`, `vimrc` merged with a `typeof stored.vimrc === "string"` check so a cleared value wins over the default like snippets, `lspMaxLines` merged with a `typeof stored.lspMaxLines === "number"` check the same way), loaded via `importScripts()` in `sw.js` and a `<script>` tag in `options.html`.
- `popup.html`/`popup.js` (`action.default_popup`) — the editor toggle, the native-mouse toggle, a "Command palette…" button, and a link to the options page. The individual per-action buttons this popup used to render (one per `SSF_TOOLS` action) now live inside the command palette instead. The palette button files-injects `editor-swap.js` (idempotent) then calls `window.__ssExt.commandPalette(libPath)` on the active tab and closes the popup.
- `options.html`/`options.js` — all configuration: a checkbox per patch (`chrome.storage.local.fixes`), a hotkey table with per-action record/clear (`chrome.storage.local.hotkeys`), an Editor section (dark/light theme selects populated from `ace/ext/themelist`, the "SAS language server" checkbox (`aceConfig.lsp`) plus a number input next to it for `aceConfig.lspMaxLines` (0 = no limit, persisted/loaded the same way as the checkbox), a "Vim config" textarea for `aceConfig.vimrc` with an explicit Save button — everything else, including the keyboard handler, is configured from Ace's own stock settings menu now and persists through the same prototype hook described above), and an Ace-editor-backed snippet text box (`chrome.storage.local.snippets.sas`, live-applied on Save via the storage-change listener in `sw.js`) that also consumes `aceConfig` (theme pair + `setOptions` + vimrc). Patch/hotkey changes apply on next page reload; snippet and editor-config changes apply immediately. Sets `ace.config.setModuleUrl` overrides for the four keyboard-handler modules (same fix as `editor-swap.js`'s `loadNewAce`) so the snippet editor can actually use them. Listens to `chrome.storage.onChanged` for `aceConfig` (e.g. a change relayed from the stock settings menu) with an `applying` re-entrancy guard so its own writes don't bounce back into itself.
- `lib/ace/` — PRISTINE upstream `ace-builds` 1.43.3 npm package (byte-identical to the npm tarball's `src-noconflict/` and `types/`), generated by `./build_lib.sh` — the single place all vendored library versions are recorded; an upgrade is a version bump there. Don't hand-edit anything under here. `ext-prompt.js` is the stock generic-prompt module the command palette is built on; `ext-settings_menu.js` is the stock settings-menu module (bundles its own `OptionPanel`) the persistence hook patches — see `editor-swap.js` above (`ext-options.js` is no longer loaded — `ext-settings_menu.js` bundles an equivalent copy of the same module under the same id).
- `src/ace-patches.js` — `window.__ssExtApplyAcePatches(ace)`, called once per loaded `ace` instance, to reproduce at runtime (each patch independently try/catch'd and idempotency-guarded) the source modifications from the author's ace fork (github.com/letuanhai/ace — commits by letuanhai only; determined by diffing that fork's commits, not by diffing the built output against upstream). It is deliberately minimal — only the fork's actual changes: (1) `ace/layer/decorators` — `$updateDecorators` fully replaced (pristine 1.43.3 body + the fork's two edits) so the scrollbar-overview cursor bar is theme-aware and every selection range is drawn as a half-width bar; (2) vim-aware Esc — `ace/autocomplete`'s `Autocomplete.prototype.commands.Esc` and `ace/snippets`'s (unexported) `TabstopManager.prototype.keyboardHandler`'s Esc both forward to the vim keyboard handler after detaching, only when vim is the active handler (`TabstopManager` isn't exported, so this grabs its prototype via a throwaway editor + snippet insertion; both modules live in `ext-language_tools`, so on pages that don't load it — the options page — the patch skips silently); (3) `ace/ext/modelist` gets a SAS entry pushed onto it (`Mode` isn't exported, so this clones an existing instance's prototype instead of `new Mode(...)`); (4) `ace/ext/statusbar` — `StatusBar.prototype.updateStatus` fully replaced with the fork's `" Line r/total (n selected), Col c "` 1-based format (skipped when `ext-statusbar` isn't loaded). Not reproduced: mouse/gutter/virtual_renderer/theme differences that showed up when diffing our old vendored build against npm 1.43.3 — those were version drift between the fork's base ace and 1.43.3, not the author's modifications, so they revert to pristine. **Call ordering matters**: `loadNewAce()` calls this after loading `ext-language_tools`/`ext-prompt`/`ext-statusbar` (which register the patched modules) but BEFORE `ext-settings_menu.js` — the latter's bundled `ace/ext/options` snapshots `modelist.modes` at load time into the settings-menu Mode dropdown, so the SAS entry must exist first. The options page does the same via `src/ace-seed.js` (a one-line `<script>` calling `__ssExtApplyAcePatches` between `ext-modelist.js` and `ext-settings_menu.js` — a file, not inline, because the extension CSP forbids inline scripts). Both the MAIN-world `AceEditorAdapter` (each SAS editor) and the options-page snippet editor construct an `ace/ext/statusbar` `StatusBar` as a non-intrusive bottom-right overlay (`pointer-events:none`), its font size tracking the configured `fontSize`; the options page loads `ext-statusbar.js` before `ace-seed.js` so the patch applies there too.
- `src/ace/` — the custom (non-upstream) Ace files that used to live under `lib/ace/`: `mode-sas.js`, `snippets-sas.js` (module id `ace/snippets/sas`), `ext-browse_ss.js` + `ext-browse_ss.d.ts` (custom file/tab-browsing extension). `editor-swap.js`'s `loadNewAce()` points `ace.config.setModuleUrl` at this directory for the `ace/mode/sas`/`ace/snippets/sas` module ids (derived from `libPath` by swapping `/lib/ace/src-noconflict` for `/src/ace`) and loads `ext-browse_ss.js` from here instead of `lib/`. Listed under `web_accessible_resources` (`src/ace/*.js`) in `manifest.json` since these load via a page-injected `<script src>`, not `chrome.scripting`. `mode-sas.js` also embeds Ace's own Python/Lua highlight rules (`require("./python_highlight_rules")`/`lua_highlight_rules")`) inside `PROC PYTHON`/`PROC LUA` `submit;…endsubmit;` blocks, so `loadNewAce()` loads `mode-python.js`/`mode-lua.js` from `lib/ace/src-noconflict/` right after `ace.js`, before any editor (or the SAS mode itself) can load.
- `lib/ace-linters/` — pristine upstream `ace-linters` 2.2.0 UMD builds (`language-client.js`, `ace-language-client.js`, copied byte-identical from the npm package's `build/` by `./build_lib.sh`; the package ships no LICENSE file, only a `package.json` `"license": "MIT"` field). The SAS language server's client half — see `editor-swap.js`'s `ensureLsp()` below.
- `lib/sas-lsp/` — the SAS language server's browser bundle. `./build_lib.sh` clones sassoftware/vscode-sas-extension at a pinned release tag, strips the embedded Pyright provider (`remove-pyright.patch` at the repo root, committed), and builds `sas-server.js` — a ~22 MB `target: webworker` bundle that self-attaches to the worker scope via `vscode-languageserver/browser` — plus `LICENSE-sas-lsp` (copied from the clone) and `.version` (the built tag, used to skip rebuilds). Like the rest of `lib/` it's gitignored generated output (the `.lsp-build/` clone dir too); `ensureLsp()` degrades gracefully (one console warning, editor works as before) if it's missing.
- The editor toggle is idempotent and repeatable — no page refresh needed to switch back and forth.

## Key SAS Studio integration points

- `window.appDMS` — main app controller; `.tabs` (SASStudioTabs), `.projects`, `.dialogs`, `.sessionId`, `.baseURL`.
- `SAS.Editor` — the editor API surface the adapter must implement (defined in `SASStudio-3.82/resources/js/sas-commons/controls/CodeEditor.js`).
- `dijit.byId(...)` / `dijit.registry` — Dojo widget access (trees are `projects.tree`, `library.tree`, `destination.tree`).
- Beware: some SAS Studio methods are subtly broken by the app itself (e.g. `dijit.Tree._expandNode` is overridden to return undefined; `ss-fixes.js` re-binds the prototype original).

## Reference documentation (read before touching the adapter)

- `SAS_EDITOR_API.md` — full `SAS.Editor` API reference (60+ methods).
- `sas-editor.d.ts` — TypeScript definitions for that API.
- `EDITOR_USAGE_MAP.md` — which SAS Studio files call which editor methods (DMSEditor is the heaviest consumer); use this to check compatibility impact of adapter changes.

## Development

No build step for the extension's own code, no Tampermonkey needed anymore; `./build_lib.sh` generates the gitignored `lib/` (run once after cloning, and after bumping a version variable at its top). Smoke test: `node test/smoke.js` — launches headless Chromium with the unpacked extension against the live instance and exercises injection, middle-click close (raw CDP input — catches event-suppression bugs synthetic dispatch can't), reopen, the native-mouse toggle, Ace activation/deactivation, the text viewer (mirroring/dirty/save/vim), the command palette (focused/unfocused/global hotkey), and the aceConfig flow (seeding, live apply, settings-menu persistence via relay.js, vimrc); see the file header for `SS_URL`/`CHROME_BIN` env vars and the playwright requirement. To test changes manually: `chrome://extensions/` → reload the unpacked extension (loaded from the repo root) → refresh the SAS Studio page → toggle with `Ctrl+.` (or the popup's toggle button). To build the publishable zip: `./package.sh` → `dist/sas-studio-ext-<version>.zip` (dist/ is gitignored; it packs `manifest.json src assets lib` — rebuilding `lib/` first if incomplete — so a new runtime file belongs in one of those). All extension logs are prefixed `[SS Ext]`; expect one line per toggle (`activated`/`deactivated ... N tab(s) ...`) plus errors/warnings only.
